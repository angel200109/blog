# eval 的功能和危害

## 简介

`eval` 能把字符串当代码执行，听起来很酷——但也仅限于"听起来"。大多数场景下用它都是个坏主意，性能和安全两样都占不了便宜。

## 核心概念

### eval 干了什么

```js
eval('console.log(1 + 1)') // 2
```

把字符串传给 JavaScript 引擎重新解析、编译、执行。正常的代码在加载时就编译好了，但 `eval` 的字符串要到运行时才能被解析，引擎做不了任何预优化。

### 明确的风险

**性能**：`eval` 阻止引擎的 JIT 优化。V8 看到 `eval` 会直接放弃对当前函数的很多优化机会——引擎不敢假设 `eval` 里面的字符串会干什么，万一它动态声明了变量、改了作用域呢？

**安全**：
```js
const userInput = 'alert("恶意代码")'
eval(userInput) // 相当于你亲手执行了用户输入的内容
```

这本质上是个 XSS 注入的后门。用户输入的内容必须永远当作文本处理，绝不能当代码执行。

**调试体验**：`eval` 里的代码没有行号、没有文件名，报错了你根本不知道具体是哪个 `eval` 出的问题。

### 有没有替代方案

- 解析 JSON：用 `JSON.parse`，不要用 `eval('(' + json + ')')`
- 动态访问属性：用 `obj[key]`，不要 `eval('obj.' + key)`
- 动态计算：能不动态就不动态，实在不行用 `new Function()`——它运行在全局作用域，至少不会污染当前作用域

## 实战场景

真实项目里基本不会主动用 `eval`。但有些老旧代码和第三方库里可能还藏着它。ESLint 里 `no-eval` 规则默认就是 error 级别。如果你确实需要动态执行代码（比如模板引擎），用 `new Function` 也比 `eval` 安全一点——至少它隔离了作用域。

## 总结

`eval` 的核心问题是把编译时机从加载推迟到了运行时，导致性能下降且无法预判副作用。能用别的方法就别碰它，这算得上 JavaScript 的最佳实践之一了。
