# JS 设计成单线程的目的

## 简介

JavaScript 从诞生之初就是单线程的，这是设计选择，不是局限。想想它最初是干什么的——操作 DOM、响应用户交互。多线程去抢同一个 DOM 节点？光想想就头疼。

## 核心概念

### 为什么是单线程

浏览器里 JS 的主要工作是：用户点了个按钮 → 改个 DOM → 绑个事件。如果是多线程，两个线程同时操作同一个 DOM 节点，浏览器不知道该听谁的，页面就直接乱套了。

你可以这样理解：多线程操作 DOM 就像两个人同时抢方向盘——不开飞的几率是零。

### 怎么处理耗时任务

单线程有一个致命问题：某段代码卡住了，整个页面就动不了。解决方案是**异步**：

```
主线程：执行同步代码 → 遇到异步任务丢给 Web API → 继续往下走
Web API：定时器/网络请求完成后 → 回调塞进任务队列
事件循环：主线程空了 → 从队列里取回调执行
```

这就是事件循环（Event Loop）的雏形。JS 本身还是单线程在跑，但浏览器是多线程的——定时器、网络请求都交给浏览器的其他线程处理，处理完把回调塞回来。

### Worker 线程

后来 HTML5 加了 Web Worker，可以开真正的多线程。但 Worker 不能操作 DOM，本质上是给主线程当苦力用的——主线程管 UI，Worker 管计算，井水不犯河水。

## 实战场景

写复杂动画或者大量数据处理时，如果不拆任务，页面就会卡。常见的做法是把大任务拆成小份用 `requestAnimationFrame` 或者丢给 Worker。React 的 Concurrent Mode 本质上也是在利用事件循环的机制，把渲染任务拆成可中断的小块。

## 总结

JS 单线程是为了省掉 DOM 操作的锁和同步问题。异步 + 事件循环是它的并发方案，Worker 是它的多线程补丁。
